## D-002 · 4 frontend-прототипа вкладки ЗОВ — выбор визуального подхода
**Дата:** 2026-06-30 · **Статус:** принято

**Контекст.** Нужно создать вкладку к смарт-процессу «Задача в отдел» (смарт #1036, плейсмент CRM_SMART_PROCESS_DETAIL_TAB). Вкладка — пульт управления ЗОВ: статус стадии, лимиты с остатками, команда, подзадачи, аккредитация, открытые проблемы. Прежде чем разрабатывать бэкенд, нужно согласовать UI.

**Варианты.**
- **V1 — Dashboard:** всё на одном экране, сетка KPI-карточек + таблица подзадач + аккредитация. Максимальная плотность.
- **V2 — Sections:** вертикальные секции сверху вниз (герой-баннер → лимиты → команда → подзадачи → аккредитация). Читается как документ.
- **V3 — Stage-flow:** визуальная цепочка стадий БП вверху, под ней 2-колоночная раскладка (лимиты + команда) и полноширинные таблицы. Акцент на текущую стадию.
- **V4 — Tabbed:** тёмная фиксированная шапка с ключевыми KPI + вкладки (Обзор / Лимиты / Команда / Подзадачи / Аккредитация). SPA-подход внутри iframe.

**Решение.** Прототипы созданы как статические HTML-файлы в `prototypes/` с реалистичными mock-данными. Выбор варианта — за заказчиком после просмотра.

**Почему.** Согласование на прототипах экономит время: переделать HTML быстрее, чем переписывать бэкенд под другой UX. Статические файлы позволяют смотреть прямо в браузере без деплоя.

**Последствия.** После выбора варианта: (1) адаптировать `www/template.html` под выбранный layout; (2) разработать PHP-эндпоинты для чтения смарта #1036, списков #75/#76/#77; (3) подключить BX24.JS для получения `entityId` текущего СП.

## D-001 · Шаблон _b24-single-php (single-tenant PHP)
**Дата:** — · **Статус:** принято

**Контекст.** Стартовали новый проект для Битрикс24.
**Решение.** PHP-бэкенд, файловый store (`<?php exit;?>`-защита), vanilla JS, Netangels.
**Почему.** Готовый скелет с OAuth, install-flow, session-gate «из коробки».
Без БД, без composer — минимум зависимостей для single-tenant.
**Последствия.** Multi-tenant / маркет — смена шаблона, не доработка этого.
