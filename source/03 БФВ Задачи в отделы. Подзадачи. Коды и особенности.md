## ⚙️Общие концепции

### 🔍 Посмотреть список пользовательских полей списка 
``` sql
SELECT * FROM b_lists_field WHERE IBLOCK_ID = 70; -- 70 это номер нашего списка
```



## 🌲История процесса. Структура списка. 
> [!note] ID списка в таблице IBLOCK_ID в таблице b_iblock_element = **70**
>  

- & В основную таблицу `b_iblock_element` **заполняем** стандартные поля как:
	- `IBLOCK_ID` = 70
	- `NAME` = **Событие** (название события)
		- Создание смарт процесса
		- Постановка задачи
		- Изучение задачи
		- Назначение ответственных
		- Выполнение задачи
		- Приемка РП
		- Приемка РГП
		- Доп информация
	- `DATE_CREATE` = NOW()
		- Пишем дату создания записи
	- `ACTIVE` = 'Y'
		- **Флаг активности** ('Y'/'N'). Можно временно скрывать элементы без удаления.
- & Забираем ключевое поле `ID`, чтобы связывать таблицу пользовательских полей с материнской
- & И дальше в польз таблицу пишем нужную информацию 
	- ~ `ID (IBLOCK_PROPERTY_ID)` в таблице `b_iblock_element_property`
		- 289 = Привязка к СП
		- 290 = Название СП
		- 291 = Содержание события
		- 297 = Код
		- 298 = Краткое название объекта
		- 302 = Задание

## Создание экземпляра смарта из под другого смарта

- Через штатные средства стандартными активити не получилось. Решил через PHP
	- ID нашего смарт процесса = 1048



``` php
use Bitrix\Main\Loader;
use Bitrix\Crm\Service\Container;

$this->WriteToTrackingService('Старт создания элемента смарт-процесса');

try
{
    if (!Loader::includeModule('crm'))
    {
        $this->WriteToTrackingService('Ошибка: модуль crm не подключен');
        return false;
    }

    $title = 'Просто обычное название';
    $entityTypeId = 1048; // ID смарт-процесса
    $assignedById = 1;    // Технический ответственный

    $this->WriteToTrackingService('TITLE=' . $title);
    $this->WriteToTrackingService('ENTITY_TYPE_ID=' . $entityTypeId);
    $this->WriteToTrackingService('ASSIGNED_BY_ID=' . $assignedById);

    $factory = Container::getInstance()->getFactory($entityTypeId);

    if (!$factory)
    {
        $this->WriteToTrackingService('Ошибка: фабрика не найдена');
        return false;
    }

    $item = $factory->createItem();

    if (!$item)
    {
        $this->WriteToTrackingService('Ошибка: не удалось создать объект элемента');
        return false;
    }

    $fields = [
        'TITLE' => $title,
        'ASSIGNED_BY_ID' => $assignedById,
    ];

    $this->WriteToTrackingService('Поля перед сохранением: ' . print_r($fields, true));

    $item->setFromCompatibleData($fields);

    $saveResult = $item->save();

    if (!$saveResult->isSuccess())
    {
        $this->WriteToTrackingService('Ошибка сохранения элемента смарт-процесса');

        $errors = $saveResult->getErrorMessages();
        if (!empty($errors))
        {
            foreach ($errors as $error)
            {
                $this->WriteToTrackingService('Save error: ' . $error);
            }
        }
        else
        {
            $this->WriteToTrackingService('Save error: ошибки не детализированы');
        }

        $this->WriteToTrackingService('Dump saveResult: ' . print_r($saveResult, true));
        return false;
    }

    $smartItemId = (int)$item->getId();

    if ($smartItemId <= 0)
    {
        $this->WriteToTrackingService('Ошибка: элемент сохранён, но ID не получен');
        return false;
    }

    $this->SetVariable('intSmartItemID', $smartItemId);
    $this->WriteToTrackingService('Смарт-процесс создан успешно, ID=' . $smartItemId);
    $this->WriteToTrackingService('Переменная БП intSmartItemID заполнена');

    return true;
}
catch (\Throwable $e)
{
    $this->WriteToTrackingService('Исключение: ' . $e->getMessage());
    $this->WriteToTrackingService('Файл: ' . $e->getFile());
    $this->WriteToTrackingService('Строка: ' . $e->getLine());
    $this->WriteToTrackingService('Trace: ' . $e->getTraceAsString());

    return false;
}
```

## Перенести факт завершения

> [!attention] Через REST API (вебхук) не срослось **по началу**.
>  Но потом через sql запросы методом научного тыка я выяснил, что нужно через вебхук править ДВА ПОЛЯ
> 1. `closedDate`
> 2. 'STATUS_CHANGED_DATE'

```batch
tasks.task.update?taskId={=Variable:intSubTaskKeyID}&fields[closedDate]={=Variable:dtFinishFact}
tasks.task.update?taskId={=Variable:intSubTaskKeyID}&fields[STATUS_CHANGED_DATE]={=Variable:dtFinishFact}
```


- Стандартное поле `closedDate` через `tasks.task.update` у вас, по сути, не меняется: документация прямо говорит, что запись и изменение полей задач выполняются **по бизнес-логике и правам**, а не все поля гарантированно доступны для произвольной записи через REST.
- Что подтверждает ваш ответ
	- Ваш запрос выполняется без ошибки, потому что `result_error` пустой, а `changedDate` и `outlookVersion` меняются, то есть обновление задачи действительно проходит. При этом `closedDate` остаётся прежним значением `2026-05-12T18:28:15+03:00`, значит сервер игнорирует попытку переписать именно это поле.
- Почему так происходит
	- В справочнике полей `closedDate` описано только как “Completion date in ISO 8601 format”, но не как поле, гарантированно доступное для ручной записи через REST. Там же есть ключевая оговорка: изменение полей зависит от бизнес-логики, роли пользователя, прав, статуса задачи и внутренних флагов, поэтому служебные поля вроде даты фактического закрытия могут рассчитываться системой и быть недоступны для прямого изменения вебхуком.[](https://apidocs.bitrix24.com/api-reference/tasks/tasks-task-update.html)
- Что это значит practically
	- Если задача уже закрыта, Bitrix24 держит `closedDate` как системную дату фактического закрытия и в стандартных отчётах опирается именно на неё; ваш лог показывает, что обычный `batch` и `tasks.task.update` это поле не переписывают даже при корректном вызове. То есть проблема не в формате `15.04.2026 13:00:00`, а в том, что стандартный REST‑метод не даёт изменить это поле на вашем портале.
- Реальный вывод
	- Через входящий вебхук и `tasks.task.update` изменить именно стандартное `closedDate` для построения штатных отчётов, скорее всего, **нельзя**. Если вам нужно именно это поле в стандартных отчётах, остаются только варианты вне обычного REST: доработка коробки на уровне PHP/модуля задач или прямое вмешательство в серверную логику, потому что штатный вебхук этот сценарий не поддерживает.


``` php
// Забираем в переменные данные из битрикса
$id = $this->GetVariable('intSubTaskKeyID'); // ID нашей подзадачи
$dtFinishFact = $this->GetVariable('dtFinishFact'); // Обновленная дата

// Работаем с БД
global $DB;  // Объект для работы с базой данных

// Закрытие задачи
$sql = "UPDATE b_tasks 
	SET STATUS = 5, 
	CLOSED_DATE = '" . $dtFinishFact . "', 
	STATUS_CHANGED_DATE = '" . $dtFinishFact . "'
	WHERE ID = " . $id;
	
// Вывожу для теста запрос в журнал
$this->WriteToTrackingService($sql, 0, 6);

// $DB->Query($sql);

//Финиш
return true;
```
